# Maintenance / 维护策略

## 中文

### 节奏现实
- 本项目是个人维护项目。
- 发布节奏是目标，不是硬性 SLA。

### 响应目标
- 普通 issue：尽量在 14 天内给出分类或状态更新。
- PR：尽量在 14 天内给出首次反馈。
- 安全问题：目标在 7 天内确认是否复现与影响范围。

### 何时会暂停
- 没有明确需求或维护兴趣时，项目可能暂停。
- 暂停不等于废弃；恢复时优先保证主线测试和文档先恢复可用。

## English

### Cadence Reality
- This is a personally maintained project.
- Release cadence is a target, not a hard SLA.

### Response Targets
- Normal issues: aim to triage or update within 14 days.
- Pull requests: aim for first feedback within 14 days.
- Security issues: aim to confirm reproducibility and impact within 7 days.

### When Development May Pause
- The project may pause when there is no concrete demand or maintenance interest.
- A pause is not the same as abandonment; when development resumes, test and doc recoverability comes first.
